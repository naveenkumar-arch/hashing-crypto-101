What is Hashing?

A hash function takes input of any size and produces a fixed-size output. It's a one-way mathematical function — fast to compute, impossible to reverse.
📥
Any Input Size
One character or one gigabyte — the hash output is always the same fixed length. "a" and the entire works of Shakespeare produce outputs of identical length.
deterministic
🔒
One-Way Only
Given a hash, you cannot compute the original input. There is no "unhash" function. This is the fundamental property that makes hashing useful for passwords.
irreversible
⚡
Avalanche Effect
Change a single character in the input and the entire hash output changes completely. "password" and "Password" produce totally different hashes.
unpredictable
❄️
Collision Resistance
Two different inputs should never produce the same hash. When this breaks (as with MD5/SHA1), the algorithm is considered cryptographically broken.
unique output

Why Linux Uses Hashing for Passwords
When you run su - alice, Linux hashes what you typed and compares it to the hash in /etc/shadow. The original password is never stored anywhere. Even root can't see it — only the hash exists. This is why cracking requires guessing forward, not reversing.
// 02

Hash Algorithms

Not all hash algorithms are equal. Some are broken, some are slow by design, some are fast by design. Knowing which is which tells you how hard it is to crack.
AlgorithmOutput SizePrefix in ShadowSpeed (GPU)Status
MD5 128-bit / 32 hex (none) or $1$ 100 billion/sec ⚠ Broken — collisions known
SHA-1 160-bit / 40 hex (none) 30 billion/sec ⚠ Broken — don't use
SHA-256 256-bit / 64 hex $5$ (sha256crypt) 10 billion/sec ✓ Safe for integrity checks
SHA-512 512-bit / 128 hex $6$ 800 million/sec ✓ Common on Linux systems
bcrypt 60 chars $2b$ / $2y$ 3,000/sec ★ Designed to be slow
yescrypt variable $y$ very slow ★ Modern Kali default
NTLM 128-bit / 32 hex (Windows SAM) 350 billion/sec ⚠ Fast — weak for passwords
⚠️
Speed is the Enemy
For passwords, slow is good. MD5 was designed to be fast (for checksums) — that's why it's terrible for passwords. bcrypt was designed specifically for passwords and is intentionally slow. A hash that takes 1ms vs 1μs means 1000x fewer guesses per second for an attacker.

Encoding vs Hashing

Encoding is NOT hashing and NOT encryption. It's a reversible transformation with no key. Anyone can decode it. You'll see it everywhere in CTFs and web hacking.
🔄
Encoding
Reversible transformation with no key. Purpose: data transport/representation, not security. Base64, hex, URL encoding, ROT13.
reversible, no key
🔒
Hashing
One-way transformation. No key needed, cannot be reversed. Purpose: integrity checking, password storage.
irreversible
🗝️
Encryption
Reversible transformation WITH a key. Without the key, you can't decrypt. Purpose: confidentiality. AES, RSA.
reversible with key
📋
The Key Test
Ask: "Do I need a key to reverse this?" No key needed → encoding. Key needed → encryption. Can't reverse → hashing.
identification

Encryption Basics

Encryption is reversible with a key. Two types: symmetric (same key to encrypt/decrypt) and asymmetric (public key encrypts, private key decrypts). You'll encounter both in CTFs.
🔑
Symmetric — AES
Same key encrypts and decrypts. Fast, used for bulk data. AES-256 is the standard. The challenge: how do you share the key securely?
one key
🗝️🔐
Asymmetric — RSA
Public key encrypts (anyone can have it). Private key decrypts (only you have it). Solves the key sharing problem. Slower than AES.
public/private pair
Encryption on KaliKALI
# AES symmetric encryption with openssl
echo "secret message" | openssl enc -aes-256-cbc -base64 -k "mypassword"
U2FsdGVkX1+... (encrypted base64 output)

# Decrypt it
echo "U2FsdGVkX1+..." | openssl enc -aes-256-cbc -d -base64 -k "mypassword"
secret message

# Generate RSA key pair
openssl genrsa -out private.pem 2048
openssl rsa -in private.pem -pubout -out public.pem

# Encrypt with public key
echo "hello" | openssl rsautl -encrypt -pubin -inkey public.pem | base64

# In CTFs: you'll often find .pem files, encrypted files
# Goal: find the private key → decrypt the file → get the flag
💡
SSH Uses Both
When you SSH into a server, it uses asymmetric RSA/ECDSA to authenticate and exchange keys, then switches to symmetric AES for the actual session data. Your ~/.ssh/id_rsa is a private key — that's why it must be chmod 600.

Salting — Why Same Password ≠ Same Hash

A salt is a random value added to a password before hashing. Even if two users have the same password, their hashes will be completely different. Defeats rainbow tables and pre-computation attacks.
// Three users, same password "password123", different salts
password123
+
$6$aBc123$
→
$6$aBc123$x9Km2pL...
password123
+
$6$xYz789$
→
$6$xYz789$mQ7nRv3...
password123
+
$6$pQr456$
→
$6$pQr456$vB2wKj9...
// The salt is stored IN the hash (after $6$) — the system reads it back to verify your password. The salt is NOT secret — its purpose is uniqueness, not secrecy.

What Salting Defeats

🌈
Rainbow Tables
Pre-computed hash→password lookup tables. Without salting, you'd just look up "5f4dcc3b..." and instantly get "password". Salting makes every hash unique so pre-computation is useless.
defeated by salt
👥
Bulk Cracking
Without salts, if two users have the same password, cracking one cracks both. With salts, you must crack each hash individually — multiplies attacker's work by number of hashes.
defeated by salt
Salting on KaliKALI
# Same password, different salts = different hashes
openssl passwd -6 -salt AAAA "password123"
$6$AAAA$hash1here...

openssl passwd -6 -salt BBBB "password123"
$6$BBBB$totallydifferenthash...

# The $6$SALT$ part is stored in /etc/shadow
# When you type password at login:
# system reads the salt from stored hash
# hashes(your_input + salt)
# compares result to stored hash

# Verify a password against its hash
python3 -c "
import crypt
stored = '\$6\$AAAA\$hash1here...'
attempt = 'password123'
print(crypt.crypt(attempt, stored) == stored)
"
True

Cracking — Full Workflow

Connecting everything together — identify the hash type, choose the right tool and attack mode, crack it, use it.
Step 1 — Identify Hash TypeKALI
# Method 1: hashid (with hashcat mode -m flag)
hashid -m '5f4dcc3b5aa765d61d8327deb882cf99'
[+] MD5 [Hashcat Mode: 0]

hashid -m '$6$salt$hash'
[+] SHA-512 Crypt [Hashcat Mode: 1800]

# Method 2: hash-identifier tool
hash-identifier
Enter hash: 5f4dcc3b5aa765d61d8327deb882cf99
MD5

# Method 3: Count characters
# 32 chars  = MD5
# 40 chars  = SHA-1
# 64 chars  = SHA-256
# 128 chars = SHA-512
# $2b$...   = bcrypt (always 60 chars)
# $6$...    = sha512crypt (Linux shadow)
Step 2 — Crack With JohnKALI
# Create hash file
echo "5f4dcc3b5aa765d61d8327deb882cf99" > hash.txt

# Auto-detect + crack
john hash.txt --wordlist=/usr/share/wordlists/rockyou.txt --format=raw-md5
password         (hash.txt)

# Show result
john hash.txt --show --format=raw-md5
?:password
Step 3 — Crack With HashcatKALI
# Same MD5 with hashcat
hashcat -m 0 hash.txt /usr/share/wordlists/rockyou.txt
5f4dcc3b5aa765d61d8327deb882cf99:password

# Shadow-style SHA-512
hashcat -m 1800 shadow_hash.txt /usr/share/wordlists/rockyou.txt
hashcat -m 1800 shadow_hash.txt /usr/share/wordlists/rockyou.txt -r /usr/share/hashcat/rules/best64.rule

# View all cracked
hashcat -m 0 hash.txt --show
